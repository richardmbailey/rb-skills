# Runtime Contract

- Source: `runtime/`; package `rb-safe-operation-runtime`; CLI module `rb_safe_operation.cli`.
- Supported Python: 3.9 or newer. Supported Pydantic: 2.12 or newer and below 3.
- The committed first-release lock includes the probed CPython 3.12 macOS ARM64 `pydantic-core` wheel hash. Other supported Python/platform combinations need a reviewed lock update and matching approved wheelhouse before setup; they fail with `copy_install_dependency_missing` rather than downloading.
- Explicit setup trust boundary: invoke `scripts/setup_runtime.py --control-root <absolute-control-root> --wheelhouse <approved-wheelhouse> --python <absolute-bootstrap-python>`. Setup records the resolved bootstrap interpreter path and SHA-256 identity in the manifest.
- Normal invocation: read only `launcher_bootstrap_interpreter_path` from the canonical `${CODEX_HOME:-~/.codex}/rb-safe-operation/current.json`, then invoke `<absolute-bootstrap-python> -I -S -B <absolute-installed-skill>/scripts/run_runtime.py <CLI command>`. The launcher has no manifest/control-root override; disposable installs use an isolated `CODEX_HOME` and explicitly set setup's destination to its `rb-safe-operation` child.
- Normal invocation ignores `RB_SAFE_OPERATION_MANIFEST` and `RB_SAFE_OPERATION_CONTROL_ROOT`; never installs, upgrades, searches sibling imports, or falls back to ambient Python. Before any subprocess it verifies bootstrap flags and identity, launcher path/hash, source and combined-lock hashes, and the installed package's observed tree hash against both its recorded identity and the manifest.
- Supported artifact schema: `1.0`. Unsupported versions and unknown fields fail closed.
- First-release active product adapters are `read_file`, `apply_patch`, plus semantically bounded agent tasks limited to `read` and `apply_patch`. `exec_argv` and `check` remain schema/dispatcher forms but fail policy until a reviewed release adds host-enforced filesystem, process, environment, and network containment; hashes alone cannot activate them, and project policy cannot widen this deny-all baseline.
- Builds, test commands, subprocesses, command-capable bounded tools, and delegation are unsupported in the first release. They stop for a revised runtime policy/typed contract rather than being smuggled through a bounded task.
- First-release assurance: Codex `semi_formal`; assessor/verifier restrictions and fresh-context separation are `instruction_only`; product comparison is `coordinator_observed`; complete child trace is unavailable. The coordinator enforces a finite `attempt_limit` when explicitly chosen and serialized request/report-byte ceilings and observes elapsed time after a host call returns. `max_calls` is per attempt and is not a cumulative retry cap; the current host cannot independently meter those internal tool calls, interrupt a blocking host call, or meter model cost, child processes, or product-file I/O. Those declared fields are external/instruction-only limits, not release-enforced meters.
- Project-local audit root only. Strict isolation, external audit roots, and Claude compatibility are unadvertised.

Use `runtime-info`, `host-capabilities`, `validate`, `canonicalize`, `hash`, `merge-policy`, `snapshot`, `select-phase`, `discover-instructions`, `assess-preflight`, `assess`, `persist-artifact`, `coordinate`, `coordinate-resume`, `render`, `export-schemas`, and `check-schema-drift`. Public `persist-artifact` creates only the fixed `low-level-plan` handoff. `assess-preflight` runs without semantic input and persists a fixed false bundle itself when deterministic checks fail; after a passing preflight, `assess` recomputes it and atomically creates the sanitized `assessment-bundle`. Both handoffs live under `.rb-safe-operation/artifacts/<run-id>/`, refuse overwrite and symlinked control roots, and keep raw agent prose transient. There are deliberately no artifact-only `execute` or caller-asserted `verify` commands. The verified `coordinate` command accepts only those exact canonical fixed paths and is a synchronous JSON-line driver: it holds the lease, emits bounded executor requests, captures product state, emits a verifier request, accepts typed agent responses, and commits the result. Fresh-context separation is instruction-only on the current host and is not labelled host-proven independence.

Before assessment, run `host-capabilities --output <outside-project-path>` and pass that unchanged file through `--capabilities`; caller-authored capability claims are rejected. Approval-gated effects use a bounded manual first-release procedure. After the human explicitly confirms one already-declared approval-class/effect/target pair, create one `approval` object outside the product snapshot for that exact pair. Bind it to the canonical low-level-plan hash, the selected operation hash, active-policy hash, and repository-snapshot hash; copy the exact effect ID/class, approval class, and target; choose a future whole-second UTC expiry; set `one_use: true`, `consumed: false`, `identity_verification: "unavailable"`, and supply an idempotency key whenever exposure, reversibility, or effect class requires one. The current host cannot authenticate a principal or mint `host_verified`; this record proves exact declared consent only at `instruction_only` identity assurance. Validate each object with `validate --artifact-type approval`, wrap the canonical objects in a JSON list, and pass the list through `--approvals`. A previously rejected bundle remains immutable: create a newly identified plan/run and pass the prior fixed bundle through `--prior-assessment-bundle`. Accepted approvals are copied into the new assessment bundle and receive a run-scoped create-only consumption receipt immediately before their operation. Ambient conversation, broad consent, duplicate IDs, expired records, reused records, and unmatched bindings never count as approval.

The manual approval channel is an instruction-only distinction, not an authenticated human boundary: an agent with permission to write the outside-project approval input can fabricate it, so the first release does not resist a mistaken or deviating equivalent-authority agent at this gate. Projects requiring authenticated approval must keep approval-gated effects at `safe: false` until a trusted host adapter can attest the principal.

## Durable coordinator API

Repository-specific mediation imports `ExecutionCoordinator` from `rb_safe_operation.workflow` inside the manifest-pinned interpreter. Constructing it creates a new run only: an existing `.rb-safe-operation/runs/<run-id>` is never reused. At each safe operation boundary it atomically persists `coordinator-bundle.json`, containing the exact typed plan, assessment, policies, capabilities, lifecycle manifest, completed report prefix, next-operation index, post-execution snapshot, and latest verification report. The manifest's event-head hash and lifecycle must match the validated append-only audit chain.

After a deliberate resource pause and lease release, use `ExecutionCoordinator.reload(<project-root>, <run-id>, <current-host-capabilities>, agent_host=...)`, then `resume_after_pause(<evidence-id>)`. Reload accepts only `paused_resource`, requires the current host capabilities to equal the assessed capabilities, reruns deterministic safety and repository-state gates, validates canonical encoding and every artifact identity, and resumes the exact `suspended_from` state. It does not replay operations already represented by the audit-committed report prefix. Repair attempts are themselves audit-committed; unaffected report prefixes may be retained, while the typed changed strategy is supplied to any bounded repair agent. Active, `human_required`, malformed, audit-divergent, and terminal runs fail closed; they must not be reconstructed as new runs under the same identifier.

The first-release `coordinate` CLI drives a new run. `coordinate-resume` reloads only a durably paused run, revalidates it, reacquires the lease, and requires a typed repair attempt when resuming `repairing`; malformed or rejected resume input returns to a durable pause or fails before lease acquisition. Both commands are stdout-only so they cannot create an undeclared product file after the final snapshot.
